## 1. 임베디드란 무엇인가

### 1.1 일상 속 임베디드 시스템

여러분의 하루를 돌아봅시다. 아침에 알람으로 깨워준 디지털 시계, 출근길에 카드를 찍은 버스 단말기, 사무실 문을 연 도어록, 점심 후 마신 자판기 커피, 퇴근길에 들른 ATM, 저녁에 돌린 세탁기, 잠자기 전 켠 IPTV의 셋톱박스. 이들 안에 모두 **컴퓨터가 들어 있습니다**. 다만 우리는 그것을 "컴퓨터"라고 부르지 않을 뿐이지요.

이렇게 **특정 목적을 수행하기 위해 다른 기기 안에 박혀 있는(embedded) 컴퓨터 시스템**을 임베디드 시스템(Embedded System)이라고 부릅니다.

### 1.2 임베디드 시스템의 정의

> **임베디드 시스템** = 어떤 기기의 일부로 내장되어, 제한된 자원(CPU·메모리·전력) 안에서 정해진 기능을 수행하도록 설계된 컴퓨터 시스템.

핵심 키워드 세 가지:

- **내장(embedded)** — 사용자가 "컴퓨터"로 인지하지 않는다. 세탁기는 세탁기이지, 누구도 "컴퓨터"라고 부르지 않는다.
- **특정 목적(special-purpose)** — 한 가지 일을 잘 해내도록 설계된다. 세탁기 컨트롤러로 게임을 하지는 않는다.
- **자원 제약(resource-constrained)** — 일반 PC보다 훨씬 적은 메모리, 적은 저장 공간, 적은 전력으로 동작한다.

### 1.3 일반 PC와 무엇이 다른가

일반 PC와 임베디드 시스템을 비교하면 다음과 같습니다.

| 항목       | 일반 PC                       | 임베디드 시스템                      |
|------------|-------------------------------|--------------------------------------|
| 사용 목적  | 다목적 (문서, 게임, 개발 등)  | 단일 목적 (예: 라우터, 셋톱박스)     |
| 사용자     | 사람이 직접 키보드/마우스로   | 사람이 의식하지 못한 채 동작         |
| 메모리     | 8 GB ~ 64 GB                  | 16 MB ~ 1 GB                         |
| 저장 공간  | SSD 256 GB ~                  | Flash 8 MB ~ 8 GB                    |
| 전력       | 50 W ~ 수백 W                 | 0.1 W ~ 수 W                         |
| OS 부팅 시간 | 30초 ~ 1분                   | 수 초 (가전·차량) ~ 0.5초 (산업)     |
| 업그레이드 | 수시로 (Windows Update 등)    | 수년에 1회 또는 거의 없음            |
| 신뢰성     | 가끔 멈춰도 됨                | 24/7 무중단이 필수인 경우가 많음     |

이 표를 통째로 외울 필요는 없습니다. 다만 "임베디드는 자원이 매우 부족하고, 부팅이 빨라야 하고, 멈추면 안 되는 환경"이라는 감각만 잡고 넘어가면 충분합니다.

## 2. 임베디드 리눅스란 무엇인가

### 2.1 한 줄 정의

> **임베디드 리눅스(Embedded Linux)** = 임베디드 시스템의 자원 제약과 목적에 맞춰 **잘게 다듬어 올린 리눅스 운영체제**.

여기서 "잘게 다듬는다"는 행위를 영어로 **Tailoring**이라고 부릅니다. 양복을 사람 몸에 맞춰 줄이고 늘리듯, 리눅스를 하드웨어와 용도에 맞춰 잘라내고 덧붙이는 작업이라고 생각하면 됩니다.

### 2.2 일반 리눅스와의 차이

같은 "리눅스"라는 이름이 붙어 있어도 데스크톱 리눅스와 임베디드 리눅스는 실제로는 매우 다릅니다.

- **이미지 크기**
  - Ubuntu Desktop: 설치 이미지 약 4 GB, 설치 후 8 ~ 15 GB 차지.
  - 임베디드 리눅스: 보통 8 ~ 64 MB. 즉 **수백 배 작다**.

- **부팅 시간**
  - Ubuntu Desktop: 30 ~ 60초.
  - 임베디드 리눅스: 1 ~ 5초가 일반적이며, 산업용은 0.5초 이하를 목표로 한다.

- **포함된 패키지**
  - Ubuntu Desktop: 수천 개 (브라우저, 워드프로세서, GUI, 게임…).
  - 임베디드 리눅스: 수십 개 (해당 제품 동작에 꼭 필요한 것만).

- **실행 위치**
  - Ubuntu: SSD/HDD에 설치되어 동작.
  - 임베디드: NAND/eMMC/NOR 같은 **플래시 메모리**에 올라가 동작. 종종 RAM 위에서만 동작(디스크리스).

- **사용자 상호작용**
  - Ubuntu: 마우스·키보드로 직접 조작.
  - 임베디드: GUI가 아예 없거나, 있어도 터치 한 가지 용도. 디버깅은 시리얼 콘솔로.

### 2.3 왜 굳이 리눅스인가

임베디드용 OS에는 RTOS, FreeRTOS, Zephyr, Nucleus 등 더 가벼운 선택지가 많은데도 임베디드 리눅스가 널리 쓰이는 이유는 다음과 같습니다.

- **무료/오픈 소스**: 라이선스 비용이 없고 소스가 공개되어 있어 어떤 커스터마이징도 가능하다.
- **풍부한 드라이버**: 거의 모든 주요 칩에 대해 누군가 이미 드라이버를 작성해 두었다.
- **강력한 네트워킹**: TCP/IP 스택이 매우 견고하다 — Wi-Fi/Ethernet 제품에 거의 표준.
- **에코시스템**: GCC, GDB, Buildroot, Yocto 같은 성숙한 툴체인이 무료로 제공된다.
- **인력 풀**: 리눅스 개발자를 구하기가 RTOS 개발자보다 훨씬 쉽다.

이 매뉴얼 시리즈는 임베디드 리눅스에 초점을 맞춥니다. RTOS와의 비교는 후속 매뉴얼에서 필요할 때 다시 다룹니다.

## 3. 핵심 용어 7가지 미리 보기

앞으로 매뉴얼 곳곳에서 반복적으로 등장할 단어들입니다. **지금 외울 필요는 없습니다.** 이 시점에 "어렴풋한 그림"만 그려 두면 충분합니다. 본격적인 사용은 해당 매뉴얼에서 다시 자세히 다룹니다.

1. **호스트(host)**
   - 여러분이 지금 사용하고 있는 **개발용 컴퓨터**.
   - 일반적으로 macOS/Windows/리눅스 PC.
   - 코드를 쓰고, 컴파일하고, 이미지를 만든다.

2. **타겟(target)**
   - 만들어진 임베디드 리눅스가 **실제로 돌아갈 기기**.
   - 예: Raspberry Pi, BeagleBone, 또는 흉내내기 환경인 QEMU.
   - 호스트와 다른 종류의 CPU(예: ARM)를 가질 때가 많다.

3. **크로스 컴파일(cross-compile)**
   - 호스트(예: x86_64 macOS)에서 **타겟(예: ARM)에서 실행될 바이너리**를 만드는 행위.
   - 우리 PC에 ARM CPU가 없는데 어떻게 ARM 프로그램을 만들지? 라는 질문의 답.

4. **툴체인(toolchain)**
   - 컴파일에 필요한 **도구 묶음**: 컴파일러(gcc), 링커(ld), 어셈블러(as), C 라이브러리, 디버거(gdb) 등.
   - 호스트에서 직접 쓰는 것이 native toolchain, 타겟용 바이너리를 만드는 것이 cross-toolchain.

5. **부트로더(bootloader)**
   - 전원을 켰을 때 가장 먼저 실행되는 **작은 프로그램**.
   - 메모리/클록을 초기화하고, 저장 공간에서 커널을 찾아 RAM에 올린 뒤 점프시킨다.
   - 임베디드에서 가장 유명한 부트로더는 **U-Boot**.

6. **커널(kernel)**
   - 운영체제의 **심장부**. 메모리 관리, 프로세스 스케줄링, 디바이스 드라이버 등 핵심 기능을 담당.
   - 우리는 커널 소스를 받아 직접 빌드하고 옵션을 조정해 볼 것이다.

7. **루트파일시스템(rootfs, root filesystem)**
   - 커널이 부팅된 뒤 마운트하는 **`/`(루트) 이하의 파일/디렉토리 모음**.
   - `/bin`, `/etc`, `/lib`, `/usr` 등이 그 안에 있다.
   - 임베디드에서는 종종 BusyBox로 최소한의 rootfs를 직접 만든다.

이 7개 용어를 한 줄로 이어보면 우리가 만들 흐름이 보입니다.

> "**호스트**에서 **크로스 컴파일러(툴체인)**를 사용해 **커널**과 **루트파일시스템**을 만들고, 이를 **부트로더**가 **타겟** 위에서 부팅한다."

이 한 줄이 임베디드 리눅스 개발의 골격입니다. 시리즈를 끝낼 즈음 이 문장이 자연스럽게 느껴질 것입니다.

- **Part IV 디바이스와 드라이버** (M011 ~)
  - 디바이스 트리, 커널 모듈, 드라이버 작성으로 들어갑니다.

```bash

```

`git version 2.x.x` 같은 출력이 나오면 설치된 상태입니다. `command not found`가 나오면 잠시 후 실습 3 안내에 따라 설치하면 됩니다.

## 6. 실습 1 — 호스트 OS와 셸 확인

이 실습의 목표는 단 하나, **자신의 호스트 환경을 정확히 파악하는 것**입니다. 앞으로의 모든 매뉴얼은 호스트 정보가 출발점이 됩니다.

### 6.1 터미널 열기

터미널 여는 방법은 OS마다 다릅니다 — 부록 A를 참고하세요. 터미널을 열면 다음과 비슷한 화면을 보게 됩니다.

```text
sl@laptop ~ %
```

이때 마지막의 `%` 또는 `$`을 **프롬프트(prompt)** 라고 부릅니다. "이제 명령을 입력해도 좋다"는 뜻입니다.

### 6.2 OS 종류 확인 — `uname`

다음 명령을 입력하고 Enter를 누릅니다.

```bash
uname -a
```

출력 예 (macOS):

```text
Darwin laptop.local 25.3.0 Darwin Kernel Version 25.3.0: ... arm64
```

출력 예 (Ubuntu):

```text
Linux ubuntu 6.5.0-25-generic #25-Ubuntu SMP ... x86_64 GNU/Linux
```

각 토큰의 의미:

- 첫 번째 토큰(`Darwin` / `Linux`) — **커널의 이름**. macOS의 커널은 Darwin, 리눅스의 커널은 Linux.
- 두 번째 토큰(`laptop.local` / `ubuntu`) — 컴퓨터의 **호스트네임**.
- 세 번째 토큰(`25.3.0` / `6.5.0-25-generic`) — 커널 버전.
- 마지막 직전 토큰(`arm64` / `x86_64`) — **CPU 아키텍처**. 이 정보가 매우 중요합니다. 앞으로 매뉴얼에서 "호스트가 x86_64인지 arm64인지"가 자주 등장합니다.

`uname -a`의 `-a`는 "all"의 약자로, 모든 정보를 한 번에 보여달라는 옵션입니다. 일부만 보고 싶다면 다음과 같이 쓸 수도 있습니다.

```bash
uname -s    # 커널 이름만
uname -m    # 아키텍처만
```

직접 한 번 실행해 보고 출력이 위 설명과 일치하는지 확인하세요.

### 6.3 셸 종류 확인

셸(shell)은 우리가 명령을 입력하면 그것을 해석해서 실행해 주는 프로그램입니다. 같은 명령이라도 셸이 다르면 동작이 미묘하게 달라질 수 있습니다.

```bash
echo $SHELL
```

출력 예:

```text
/bin/zsh
```

또는

```text
/bin/bash
```

`zsh`(Z 셸)와 `bash`(Born Again Shell)가 가장 흔한 두 셸입니다. 이 매뉴얼의 모든 명령은 둘 다에서 동작합니다 — 차이가 발생하면 그 자리에서 따로 안내합니다.

`$SHELL`처럼 `$`로 시작하는 단어는 **환경 변수(environment variable)** 입니다. 시스템이 미리 설정해 둔 값이라고 이해하면 됩니다.

### 6.4 정리

여기까지 따라 했다면 여러분은 다음 정보를 알게 되었습니다.

- 내 OS(커널 이름)는 무엇인가.
- 내 CPU 아키텍처는 무엇인가 (`x86_64` 또는 `arm64`).
- 내 셸은 무엇인가 (`zsh` 또는 `bash`).

이 세 가지를 메모해 두세요. 다음 매뉴얼(M002)부터 자주 묻습니다.

## 7. 실습 2 — 학습 디렉토리 만들기

이 시리즈를 진행하면서 만들 파일이 점점 늘어납니다. 한 곳에 모아두면 잃어버릴 일이 없습니다.

### 7.1 홈 디렉토리 확인

홈 디렉토리는 사용자 본인의 "내 폴더"입니다. 보통 macOS는 `/Users/<이름>`, Linux는 `/home/<이름>`입니다.

현재 위치를 확인합니다.

```bash
pwd
```

출력 예:

```text
/Users/sl
```

`pwd`는 "print working directory"의 약자입니다. 만약 다른 위치에 있다면 홈으로 돌아갑니다.

```bash
cd ~
```

`~`(틸드)는 "현재 사용자의 홈 디렉토리"를 의미하는 약식 표기입니다. `cd`는 "change directory"입니다.

### 7.2 디렉토리 생성

학습 전용 디렉토리 `embedded-linux-study`를 홈에 만듭니다.

```bash
mkdir -p ~/embedded-linux-study
```

옵션과 인자:

- `mkdir` — "make directory"의 약자. 디렉토리를 생성한다.
- `-p` — "parents"의 약자. 부모 디렉토리가 없으면 함께 만들고, 이미 디렉토리가 있어도 에러를 내지 않는다. 같은 명령을 여러 번 실행해도 안전하다.
- `~/embedded-linux-study` — 만들 경로. `~`은 홈 디렉토리.

실행하고 나면 아무런 출력이 없습니다 — 유닉스 도구는 보통 **성공하면 아무 말도 하지 않습니다**. "no news is good news" 원칙입니다.

### 7.3 진입 및 확인

```bash
cd ~/embedded-linux-study
pwd
ls -la
```

각각의 의미:

- `cd ~/embedded-linux-study` — 방금 만든 디렉토리로 이동.
- `pwd` — 현재 위치를 다시 확인 (`/Users/sl/embedded-linux-study` 등).
- `ls -la` — 현재 디렉토리의 모든 파일을 자세히(`-l`) 그리고 숨김 파일까지(`-a`) 보여준다.

`ls -la` 출력 예:

```text
total 0
drwxr-xr-x   2 sl  staff   64 May  4 03:00 .
drwxr-xr-x  50 sl  staff 1600 May  4 03:00 ..
```

여기서 `.`은 현재 디렉토리, `..`은 상위 디렉토리를 가리키는 특별한 항목입니다. **아직 우리가 만든 파일은 없는 빈 디렉토리** 상태입니다.

## 8. 실습 3 — git으로 학습 기록 시작

git은 파일의 변경 이력을 추적하는 **버전 관리 도구**입니다. 학습 기록을 git으로 관리하면 "어제는 무엇까지 했더라?"를 언제든 되짚을 수 있습니다.

### 8.1 git 설치 확인

```bash
git --version
```

`git version 2.x.x`가 나오면 OK. 만약 `command not found`가 나온다면:

- macOS: 터미널에 `xcode-select --install` 입력 → 안내 창에서 "설치" 클릭.
- Ubuntu/Debian: `sudo apt update && sudo apt install -y git` 실행.
- Windows: <https://git-scm.com/download/win> 에서 Git for Windows 설치.

설치 후 다시 `git --version`을 확인합니다.

### 8.2 git 사용자 정보 설정 (최초 1회)

git은 누가 변경했는지를 모든 커밋에 기록합니다. 컴퓨터 전역으로 한 번만 설정하면 됩니다.

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

따옴표 안의 값은 본인의 이름과 이메일로 바꿔서 입력하세요. 이미 설정되어 있다면 다시 할 필요는 없습니다. 현재 설정을 확인하려면:

```bash
git config --global --list
```

`user.name=...`과 `user.email=...` 줄이 보이면 OK.

### 8.3 저장소 초기화

학습 디렉토리 안에서 git 저장소를 시작합니다.

```bash
cd ~/embedded-linux-study
git init
```

출력 예:

```text
Initialized empty Git repository in /Users/sl/embedded-linux-study/.git/
```

방금 만들어진 `.git` 디렉토리가 git의 모든 이력을 저장하는 곳입니다. 절대로 직접 수정해서는 안 됩니다.

```bash
ls -la
```

이번에는 `.git` 디렉토리가 보일 것입니다.

```text
total 0
drwxr-xr-x   3 sl  staff   96 May  4 03:05 .
drwxr-xr-x  50 sl  staff 1600 May  4 03:00 ..
drwxr-xr-x   9 sl  staff  288 May  4 03:05 .git
```

### 8.4 첫 메모 작성

학습 노트로 사용할 `NOTES.md` 파일을 만듭니다. 다음 명령은 셸에서 직접 짧은 텍스트를 파일로 흘려보내는 방식입니다.

```bash
cat > NOTES.md <<'EOF'
# Embedded Linux 학습 노트

## Manual 001 완료
- 임베디드/임베디드 리눅스 개념 학습
- 핵심 용어 7가지 미리 보기
- 학습 디렉토리 생성 및 git 초기화 완료

## 메모
- 내 호스트 OS:
- 내 호스트 아키텍처:
- 내 셸:
EOF
```

실행 후 어떤 일이 일어났는지:

- `cat > NOTES.md` — `cat`이 받은 입력을 그대로 `NOTES.md`라는 이름으로 저장.
- `<<'EOF' ... EOF` — 이 사이에 끼어 있는 모든 줄을 입력으로 사용. `EOF`는 "End Of File"의 약자로, 흔히 쓰이는 마커일 뿐 다른 단어로 바꿔도 됩니다(시작과 끝이 같기만 하면 됨).

작성된 파일을 확인합니다.

```bash
cat NOTES.md
```

방금 입력한 내용이 그대로 출력되어야 합니다. 빈 칸(`내 호스트 OS:` 옆 등)은 본인이 텍스트 에디터로 채워 넣어도 좋습니다 — VS Code 사용자라면 다음 명령으로 열 수 있습니다.

```bash
code NOTES.md
```

(VS Code가 없거나 다른 에디터를 쓴다면 그냥 건너뛰어도 됩니다.)

### 8.5 첫 커밋

git에 변경 사항을 기록하는 것을 **커밋(commit)** 이라고 합니다. 커밋은 두 단계입니다.

1. **add** — 변경 사항을 "다음 커밋에 포함시킬 후보"로 등록(스테이징).
2. **commit** — 후보를 실제 이력에 새겨 넣음.

```bash
git status
```

출력 예:

```text
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        NOTES.md

nothing added to commit but untracked files present (use "git add" to track)
```

`Untracked files` — 아직 git이 관리하지 않는 파일이라는 의미입니다. add 해 줍니다.

```bash
git add NOTES.md
git status
```

이제 출력은 다음과 같이 바뀝니다.

```text
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   NOTES.md
```

`Changes to be committed` 안에 들어왔습니다. 이제 커밋합니다.

```bash
git commit -m "Manual 001 완료 - 학습 노트 시작"
```

`-m`은 커밋 메시지를 명령행에서 바로 주는 옵션입니다. 출력 예:

```text
[main (root-commit) a1b2c3d] Manual 001 완료 - 학습 노트 시작
 1 file changed, 11 insertions(+)
 create mode 100644 NOTES.md
```

`(root-commit)`은 **이 저장소의 첫 번째 커밋**이라는 표시입니다. 이력을 확인합니다.

```bash
git log
```

출력 예:

```text
commit a1b2c3d... (HEAD -> main)
Author: Your Name <you@example.com>
Date:   Mon May 4 03:10:00 2026 +0900

    Manual 001 완료 - 학습 노트 시작
```

`q`를 누르면 빠져나옵니다 (`git log`는 페이저에서 열림).

축하합니다 — 임베디드 리눅스 학습 여정의 첫 커밋이 만들어졌습니다.

## 9. 코드 미리보기 — `hello.c` (지금은 보기만)

본격적인 컴파일은 M004부터 다루지만, 우리가 결국 어떤 코드를 만지게 될지 미리 한번 봐 둡시다.

```c
#include <stdio.h>

int main(void) {
    printf("Hello, Embedded Linux!\n");
    return 0;
}
```

한 줄씩 읽어 봅니다.

- `#include <stdio.h>` — "표준 입출력(standard input/output)" 라이브러리를 불러온다. `printf`를 쓰기 위해 필요.
- `int main(void)` — 프로그램의 시작 지점. 모든 C 프로그램은 `main` 함수에서 실행이 시작된다. `int`는 종료 코드를 정수로 반환한다는 뜻, `(void)`는 인자를 받지 않는다는 뜻.
- `{ ... }` — 함수의 본문(body)을 감싸는 중괄호.
- `printf("Hello, Embedded Linux!\n");` — 따옴표 안의 문자열을 화면에 출력. `\n`은 줄바꿈 문자.
- `return 0;` — `0`을 반환. 유닉스 관습에서 `0`은 "정상 종료"를 의미. 다른 값은 "에러"로 해석된다.
- 마지막 `}` — `main` 함수의 끝.

지금 단계에서는 **타이핑하거나 실행할 필요가 없습니다.** M004에서 똑같은 코드를 다시 등장시켜 직접 컴파일하고, 호스트와 ARM 양쪽에서 실행해 볼 것입니다. 다만 이 6줄짜리 코드가 우리 학습의 출발점이라는 사실은 기억해 둡시다.

## 10. 자주 묻는 질문 (FAQ)

**Q1. macOS 사용자인데 임베디드 리눅스 학습이 가능한가요?**

가능합니다. 이 매뉴얼은 호스트 OS를 가리지 않습니다. 다만 본격적인 커널 빌드 단계(M009)에서는 리눅스 환경이 필요한데, 이는 M002에서 가상머신 또는 컨테이너로 마련합니다.

**Q2. 임베디드 보드(Raspberry Pi 등)를 지금 사야 하나요?**

아니요. M001 ~ M010은 모두 PC만으로 진행합니다(QEMU라는 에뮬레이터를 사용). 실제 보드는 Part V에서 다룹니다. 그때 가서 사도 늦지 않습니다.

**Q3. C 언어를 모르는데 따라갈 수 있을까요?**

기초 수준은 따라갈 수 있도록 코드 한 줄씩 풀이를 곁들입니다. 다만 깊게 파고들 때를 대비해 `printf`/`scanf`/포인터 정도의 기본은 별도로 익혀두면 좋습니다.

**Q4. 윈도우에서 그냥 PowerShell만 써도 되나요?**

이 매뉴얼의 명령은 대부분 PowerShell에서도 동작합니다(`uname`은 동작하지 않을 수 있어요). 그러나 M002 이후 리눅스 환경이 필요해지므로, 가능하면 **WSL2(Windows Subsystem for Linux)** 사용을 권장합니다. 자세한 안내는 M002에서.

**Q5. 한 번에 다 외워야 하나요?**

아니요. 매뉴얼은 **참조(reference)** 입니다. 필요할 때 다시 펼쳐 보면 되고, 시간이 흐르면 자주 쓰는 명령은 자연스럽게 손에 익습니다.
